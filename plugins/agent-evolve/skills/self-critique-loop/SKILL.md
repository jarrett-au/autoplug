# 自省迭代循环 (Self-Critique Loop)

不带偏见、不降低标准地迭代产出，直到达到真实的 "aha，这就是我想要的"时刻。

## 适用场景

- **文档迭代**：写完一版完整文档 → 从目标受众视角审视 → 改进 → 重复
- **方向探索**：提出一个想法/立意 → 不断追问"这真的是我想要的吗？" → 深挖直到 aha
- **方案打磨**：给出一个方案 → 从用户/客户/专家视角挑刺 → 修改 → 重复

## 核心原则

1. **不降低标准**：不因为"已经改了三轮了"就放过问题。不满意就继续
2. **不带偏见**：不预设"这个方向是对的"，每个论点都要经得起审视
3. **完整产出**：每次迭代都产出完整版本，不是补丁。宁可重写也不要碎片修补
4. **诚实批判**：假装自己是目标受众/用户，真实地感受"这段让我想听吗？这里我有共鸣吗？"
5. **aha 才停**：不是"还行"就停，是"对，这就是我要的"才停。如果内心还有"差一点"的感觉，继续

## 执行流程

### 第零步：预先定义 aha 标准（关键）

**在写任何东西之前，先定义"怎样才算 aha"。** 不要事后才判断——预先写下 3-5 条可检验的标准。

为什么？事后判断容易"差不多就觉得行"（premature convergence）。预先定义标准后，每轮迭代都用同一把尺子量，不容易自我麻痹。

**aha 标准示例（分享文档场景）：**

1. **认知翻转**：读者读到某处时，大脑中的"理所当然"被翻转。检验：一句话概括这个翻转，没读过文档的人也会产生"等一下"的反应
2. **具体可锚定**：翻转指向读者明天就能体验的场景。检验：读者能立刻想到"上周四那个情况就是这个"
3. **不可逆**：读过之后无法回到原来的认知方式。检验：读完后再看"AI 不行"这个判断，会觉得这个判断本身就有问题
4. **黄金比例**：目标受众（如产品+开发混合）中，80% 的人能跟上每个概念。检验：每个技术概念是否都有一个"一句话大白话"的锚点

**原则：标准必须具体到可以"检验"（pass/fail），不能模糊（"有深度""有共鸣"不是标准）。**

### 第一步：明确审视视角 + 素材先行

在开始 loop 之前，完成两件事：

**A. 确定审视视角（"我是谁"）：**

- **受众视角**：我是坐在这场分享的观众，我能听懂吗？我会走神吗？哪段让我坐直身体？
- **用户视角**：我是这个产品的用户，这个功能解决我的痛点了吗？我会用吗？
- **专家视角**：我是这个领域的专家，这里有硬伤吗？逻辑链完整吗？
- **自我视角**：我是我自己，这真的是我想表达的吗？还是我在迎合想象中的期待？

可以组合多个视角，但每次迭代至少用一个固定视角走完全程。

**B. 素材先行（如果任务涉及内容创作）：**

先完整阅读/加载所有参考素材，再开始写。不要边写边查——这会导致"手头有什么写什么"而非"素材中什么最有洞察力"。

具体做法：
1. 列出所有素材来源
2. 逐个阅读/加载，记录每个素材中的"意外事实"和"反直觉洞察"
3. 从素材中提炼候选立意/主题（不要预先决定，让素材说话）
4. 用预先定义的 aha 标准筛选候选立意
5. 确认立意后，才开始写完整文档

**素材先行的反模式（避免）：**
- 预先决定立意 → 找素材来支撑 → 结果是"论证预设结论"而非"发现真正洞察"
- 边写边查素材 → 结果是被先写的段落限制了后面的方向

### 第二步：Loop 循环

每次迭代的固定节奏：

```
while (not aha):
  ┌─────────────────────────────────────────┐
  │  1. 产出完整版本                         │
  │     （不是补丁，是完整重写或大幅重构）      │
  │                                         │
  │  2. 换位审视                             │
  │     （用确定的视角，像那个人一样读一遍）     │
  │                                         │
  │  3. 双标准批判                           │
  │                                         │
  │  A. 用户需求清单（必须有明确的用户反馈/    │
  │     要求时使用，没有则跳过）：              │
  │     - 逐条核对用户提过的每一条要求         │
  │     - 标记 ✅ 已满足 / ❌ 未满足           │
  │     - 未满足的 → 记录具体差距和改进方案     │
  │                                         │
  │  B. 质量/aha 标准清单：                   │
  │     - 用预先定义的 aha 标准逐条检验        │
  │     - 用通用批判清单逐维度检查              │
  │     - 标记每条通过/未通过                  │
  │                                         │
  │  C. 改进计划：                           │
  │     - 从 A+B 中提取必须改的问题            │
  │     - 按影响大小排序（立意问题 > 结构 >   │
  │       段落 > 措辞）                        │
  │     - 判断：patch 还完整重写？             │
  │                                         │
  │  4. 防倦怠检查（每 3 轮强制执行）         │
  │     - 换视角？重新看最差版？方向是不是     │
  │       走偏了？收敛于平庸局部最优了吗？     │
  │                                         │
  │  5. 判定                                 │
  │     - 如果 aha → break, 输出最终版       │
  │     - 如果还差一点 → 记录问题, continue   │
  └─────────────────────────────────────────┘
```

**双标准批判的要点**：用户需求清单确保"没忘用户要求什么"，质量标准清单确保"没降低自己的判断力"。两个清单独立打分——不能因为用户要求都满足了就觉得"aha"（用户没要求的地方可能有问题），也不能因为"自己觉得好"就忽略用户明确提过的反馈。两条线都通过才算 aha。

### 第三步：记录迭代过程

每次迭代用统一的格式记录：

```
Loop N 自评：

视角：[用的哪个视角]

✅ 好的：
- [具体哪部分好，为什么]

❌ 问题：
- [具体哪部分不好，为什么，怎么改]

🤔 犹豫：
- [不确定的地方，需要下一轮验证的想法]

判定：继续 / ✅ aha
```

这个记录有两个价值：
- 让迭代过程可追溯，避免"改了两轮忘了第一轮的问题"
- 让用户看到你的思考过程，理解为什么最终版是这样的

## 批判清单（通用）

每次审视时，依次检查这些维度：

| 维度 | 问题 |
|------|------|
| **共鸣** | 开头 30 秒能抓住目标受众吗？他们会有"对，就是这样"的感觉吗？ |
| **结构** | 信息流的顺序合理吗？有没有"到这里突然不知道在讲什么"的断裂？ |
| **节奏** | 有没有"讲了很久但没推进"的拖沓？有没有"太快没交代清楚"的跳跃？ |
| **深度** | 是只说了"是什么"，还是也说了"为什么"和"跟我有什么关系"？ |
| **冲击** | 有没有一个让人"坐直身体"的时刻？如果没有，差在哪里？ |
| **行动** | 读完之后，受众知道该做什么吗？还是只有"有道理"但没有下一步？ |
| **收尾** | 结尾和开头呼应了吗？有没有留下一句值得记住的话？ |

## 退出条件：while(aha) loop

**核心规则：不预设轮次上限。aha 才停，否则永远保持审慎。**

这不是"最多跑 5 轮"——这是一个 while 循环。什么时候 aha，什么时候退出。第 3 轮 aha 可以，第 12 轮才 aha 也可以。唯一的退出条件是 genuine aha。

**Genuine aha 的特征（三个必须同时满足）：**

1. **内心确认**：从头到尾读一遍，没有"这里我想跳过"或"差一点"的冲动。如果现在就交付，你不会觉得"再改改就好了"
2. **逻辑闭合**：每个部分都知道"为什么在这里"，没有悬而未决的问题，没有"这个之后再补"的遗留
3. **视角验证通过**：用目标受众视角走完全程，没有"到这里我会走神"或"这段跟我没关系"的时刻

**不是 aha 的特征（继续迭代）：**

- "还行"、"差不多"、"可以了"——这些都不是 aha
- "再改一轮应该就好了"——那就再改一轮
- "这个问题不大"——那就把它修好
- "用户可能不会注意到"——如果你自己注意到了，那就改
- "我觉得已经很好了"——好和 aha 之间有一道鸿沟。好是及格线，aha 是"对，就是这个"

## 防倦怠机制

长时间迭代容易自我麻痹（"改了好几轮了，应该差不多了"）。用以下方法保持锐度：

- **换视角重审**：如果同一视角审了 3 轮还在原地，换一个完全不同的视角（受众→专家→对手→自我）
- **重新阅读最差的那一版**：找出一版最差的，读一遍。你可能会重新发现之前觉得"已经修好了"的问题其实还在
- **假设是错的**：每 3 轮主动问一次"整个方向是不是走偏了？"，然后诚实回答。如果是，推翻重来，不要因为沉没成本维护一个错误方向
- **暂停检查**：如果连续 2 轮的改动越来越小（措辞微调），可能是"小修"而非"大改"——停下来问：是真的接近 aha 了，还是我在收敛于一个平庸的局部最优？

## 注意事项

- **不要 prematurely converge**：觉得"不错"的时候最容易停，但"不错"和 aha 之间往往还有一到两次质变
- **不要怕推翻前面的方向**：如果审视中发现方向本身有问题，果断推翻重来。沉没成本是最贵的成本
- **区分"小修"和"大改"**：如果只是措辞和顺序问题，打 patch；如果结构或立意有问题，完整重写
- **改进优先级排序**：立意/核心论点 > 整体结构 > section 间过渡 > 单个段落 > 措辞微调。先修高层问题，低层问题可能随高层修复自动解决
- **向用户报告进度**：每完成一轮 loop，简要报告改动和判定结果。如果用户中途提出方向调整，立即纳入下一轮迭代
- **警惕功能列表陷阱**：当文档中段开始逐个介绍功能/特性（"A 是什么""B 是什么""C 是什么"），说明主线在断裂。每个功能都必须绑回核心立意——如果不绑回，砍掉。读者的注意力预算有限，每个段落都必须为 aha 服务，而不是在"展示我知道得多"
- **事前标准 > 事后感觉**：预先定义的 aha 标准比迭代中的"我觉得差不多了"更可靠。如果事前标准没通过，不要因为"改了好几轮了"就放过
