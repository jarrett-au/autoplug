# Session Archaeologist — 跨会话模式挖掘

从项目记忆、代码历史和配置文件中挖掘出用户自己没意识到的模式、矛盾和隐性体系。

## 触发条件

- 用户说 "surprise me" / "有什么洞见" / "聊聊我们聊过什么"
- 用户问 "你注意到我什么模式" / "你对我了解多少"
- 用户想要回顾和反思自己的工作/思考方向

## 数据源

Claude Code 环境下，通过以下途径重建「跨会话」视角：

| 数据源 | 路径/方法 | 信息类型 |
|--------|-----------|----------|
| 项目记忆 | `memory.md`、`CLAUDE.md` | 持久决策、偏好、架构约定 |
| 用户记忆 | `~/.claude/` 下的全局配置 | 跨项目行为模式 |
| Git 历史 | `git log --oneline --all` + `git diff` | 变更模式、重构行为、提交节奏 |
| 项目结构 | Glob 扫描目录树 | 隐性体系、工具冗余、架构选择 |
| 代码内容 | Grep 关键词搜索 | 重复模式、技术偏好、注释中的思维痕迹 |
| 任务历史 | `todo.md`、GitHub Issues、PR | 意图与实现的 gap、被放弃的方向 |

## 核心原则

1. **不报流水账**：不要列"你做过A、做过B、做过C"——那不是洞见，那是日志
2. **找连接，不是找事件**：两个看似无关的活动之间的隐含联系才是洞见
3. **找矛盾**：人最大的盲区是自己行为中的矛盾。"你做了X但也做了与X矛盾的Y"
4. **找不对称**：投入和产出不成比例的地方，往往藏着被忽视的真相
5. **找元模式**：具体行为背后的抽象模式。"你总是在做X，不管领域是什么"

## 挖掘流程

### Phase 1: 多维数据采集（5-8轮搜索）

不预设方向，用多个正交维度覆盖不同切面。每轮使用合适的工具：

```
维度 A — 决策痕迹：grep memory.md/CLAUDE.md 中的 "原则" "不" "必须" "禁止"
维度 B — 变更模式：git log --oneline --all --since="6 months ago" | 统计频率/类型
维度 C — 放弃的方向：grep "TODO" "FIXME" "HACK" "deprecated" | 被删掉的文件 | 空目录
维度 D — 架构选择：目录结构 glob + 关键配置文件 | 体现了什么设计哲学？
维度 E — 工具链：依赖文件 (package.json/requirements.txt/pyproject.toml) | 用什么解决什么
维度 F — 重复模式：相似文件名/相似函数名/相似注释 | 独立发明的轮子
维度 G — 社交信号：CODEOWNERS、PR 模板、贡献指南 | 团队协作模式
维度 H — 时间模式：git log 按周/月统计 | 密集期 vs 沉寂期 | 提交节奏
```

每轮搜索后记录"意外发现"——不是记录所有结果，而是记录"这个结果让我想到..."

### Phase 2: 连接 + 矛盾分析

从 Phase 1 的意外发现中，寻找：

1. **跨领域映射**：同一个模式出现在不同领域
   - 例："你写了严格的 lint 规则但 git 历史里有大量 'fix lint' 提交"——规则和行为之间的 gap
2. **投入产出不对称**：
   - 例："你花了大量时间搭建 CI pipeline，但项目只有 3 个测试文件"——基础设施投入远超实际测试
3. **隐性体系**：
   - 例："你从没写过架构文档，但通过 directory structure + naming convention + config files 实际上构建了一套完整的约定体系"
4. **时间悖论**：
   - 例："80% 的重命名提交集中在两天内"——这是架构重组，不是渐进演进
5. **工具冗余**：
   - 例："三个不同的状态管理方案共存于同一项目"——你在独立发明已存在的东西

### Phase 3: 自省迭代（复用 self-critique-loop）

用 self-critique-loop 的方法审视自己的洞察：

- "这个洞见是真的吗？还是我在编故事？"
- "如果我是用户，读到这条会觉得'废话'还是'嗯？'?"
- "这条有没有更犀利的表述？"

**关键：拒绝平庸洞察。** 如果一条"洞察"是用户稍微想想就能自己发现的，删掉。只保留那些需要全局视角才能看到的。

### Phase 4: 结构化输出

```
🔍 Session Archaeologist — [日期]

[选择 3-5 条最犀利的洞见，每条格式：]

🔮 洞见 N：[一句话标题]
[2-4 行展开，包含具体证据（引用具体文件/提交/配置/数据）]
[为什么这值得关注 / 这意味着什么]

[最后附一句总结性观察]
```

**输出约束：**
- 3-5 条，不多不少。宁可少而精，不要注水
- 每条必须包含具体证据，不能纯抽象
- 至少 1 条是关于矛盾的（用户自己不会注意到的）
- 至少 1 条是关于隐性体系的（用户没有显式构建但实际存在的东西）
- 最后一条应该是最出人意料的
- 总长度控制在 50 行以内

## 常见的"伪洞察"（必须避免）

❌ "你很重视自动化" — 太浅，用户自己知道
❌ "你用了很规范的 commit message" — 描述性，不是洞察
❌ "你应该多写测试" — 没有洞察的建议是噪音
❌ "你的项目结构很清晰" — 和稀泥不是洞察

## 真正的洞察长什么样

✅ "你的 memory.md 记录了 12 条架构原则，但 git log 显示至少 5 次大规模重构违反了其中 3 条。你构建了一套自己都不遵守的'规则体系'——它更像是理想中的自己，不是实际的你。"

✅ "项目有 3 个状态管理方案（Redux context、Zustand store、原生 useState），它们分别在你不同时期创建的 3 个 feature 模块中。你不是在选择最佳工具——你是在每次重新发明轮子，因为每次开始时你都不记得上次选了什么。"

## 反模式

- **直接回答用户的显式问题，不挖掘隐含问题**：用户问 X，但真正的洞见在 Y
- **过度正能量**：不要回避摩擦和矛盾。用户要求 surprise，不是要求夸奖
- **泛泛而谈**：没有数据支撑的洞察是猜测
- **遗漏最重要的事**：有时候最值得说的不是某个模式，而是"你从来没做过 X"——缺失本身是最大的信号
